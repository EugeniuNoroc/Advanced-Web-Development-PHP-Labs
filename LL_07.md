## Шаг 1. Нативные PHP-шаблоны

### Принцип разделения

До рефакторинга в одном файле находилась и логика (запросы, валидация, сортировка) и HTML-разметка. После разделения:

- **Логика** (`index.php`, `form.php`, `delete.php`) — только обработка данных, подготовка переменных, редиректы.
- **Представление** (`templates/native/*.php`) — только вывод HTML, никаких обращений к базе или файлам.

### Функция `render()`

Для передачи данных из логики в шаблон реализована функция буферизованного рендеринга в `src/render.php`:

```php
/**
 * Рендерит нативный PHP-шаблон, оборачивая его в общий макет.
 *
 * @param string $template Имя шаблона (без пути и расширения).
 * @param array  $vars     Переменные, доступные в шаблоне.
 * @return void
 */
function render(string $template, array $vars = []): void
{
    extract($vars);        // превращает ключи массива в переменные

    ob_start();            // включаем буферизацию вывода
    require __DIR__ . '/../templates/native/' . $template . '.php';
    $content = ob_get_clean(); // захватываем вывод шаблона в $content

    require __DIR__ . '/../templates/native/layout.php'; // вставляем в макет
}
```

`ob_start()` и `ob_get_clean()` — механизм буферизации вывода. Вместо немедленного вывода в браузер PHP копит всё в буфер, затем `ob_get_clean()` забирает его содержимое как строку. Эта строка передаётся в `layout.php` через переменную `$content`.

### Общий макет `layout.php`

Содержит HTML-обёртку, навигацию, стили и выводит `$content` в нужном месте:

```php
// templates/native/layout.php
<h1>🔭 SkyLog — Дневник наблюдений</h1>
<nav>...</nav>

<?php if (!empty($flash)): ?>
    <div class="alert alert-<?= htmlspecialchars($flash['type']) ?>">
        <?= htmlspecialchars($flash['msg']) ?>
    </div>
<?php endif; ?>

<?= $content ?>   <!-- сюда вставляется результат шаблона list.php или form.php -->
```

### Вызов из логики

```php
// index.php — только подготовка данных
$observations = $storage->loadAll();
usort($observations, ...);

render('list', [
    'title'        => 'SkyLog — Список наблюдений',
    'observations' => $observations,
    'sort'         => $sort,
    'dir'          => $dir,
    'flash'        => $flash,
]);
```

## Шаг 2. Twig

### Установка

Twig устанавливается через Composer. В корне проекта создан файл `composer.json`:

```json
{
    "require": {
        "twig/twig": "^3.0"
    }
}
```

Установка выполняется командой в терминале:

```bash
composer install
```

После этого появляется папка `vendor/` с библиотекой и файл `vendor/autoload.php` для автозагрузки.

### Инициализация в `src/twig.php`

```php
function createTwig(): \Twig\Environment
{
    $loader = new \Twig\Loader\FilesystemLoader(__DIR__ . '/../twig-templates');
    $twig   = new \Twig\Environment($loader, ['cache' => false]);

    // регистрация фильтров...

    return $twig;
}
```

`FilesystemLoader` — говорит Twig где искать файлы шаблонов. `Environment` — основной объект Twig, через который вызывается рендер.

### Наследование шаблонов

Главное преимущество Twig над нативными шаблонами — наследование. Базовый макет `layout.twig` объявляет блок:

```twig
{# layout.twig #}
{% block content %}{% endblock %}
```

Дочерние шаблоны расширяют макет и заполняют блок:

```twig
{# list.twig #}
{% extends 'layout.twig' %}

{% block content %}
    <div class="card">
        ...содержимое страницы...
    </div>
{% endblock %}
```

`{% extends %}` — говорит что этот шаблон наследует другой. `{% block content %}` — определяет какую часть макета заменить. Всё что снаружи блока — берётся из родителя автоматически.

### Синтаксис Twig vs нативный PHP

| Действие | Нативный PHP | Twig |
|---|---|---|
| Вывод переменной | `<?= $var ?>` | `{{ var }}` |
| Условие | `<?php if (...): ?>` | `{% if ... %}` |
| Цикл | `<?php foreach (...): ?>` | `{% for item in items %}` |
| Фильтр | `htmlspecialchars($v)` | `{{ v \| e }}` |
| Комментарий | `<?php // ... ?>` | `{# ... #}` |

Twig автоматически экранирует все переменные при выводе через `{{ }}` — не нужно вручную писать `htmlspecialchars()` везде.

## Шаг 3. Кастомные фильтры Twig (+1 балл)

Зарегистрированы два собственных фильтра в `src/twig.php`:

### Фильтр `format_conditions`

Переводит техническое значение условий наблюдения в читаемое название:

```php
$twig->addFilter(new \Twig\TwigFilter('format_conditions', function (string $conditions): string {
    return match ($conditions) {
        'excellent' => 'Отличные',
        'good'      => 'Хорошие',
        'fair'      => 'Удовлетворительные',
        'poor'      => 'Плохие',
        default     => $conditions,
    };
}));
```

Использование в шаблоне:

```twig
{{ obs.conditions | format_conditions }}
{# 'good' → 'Хорошие' #}
```

### Фильтр `format_type`

Переводит тип астрономического объекта на русский язык:

```php
$twig->addFilter(new \Twig\TwigFilter('format_type', function (string $type): string {
    return match ($type) {
        'planet'  => 'Планета',
        'star'    => 'Звезда',
        'galaxy'  => 'Галактика',
        'nebula'  => 'Туманность',
        'meteor'  => 'Метеор',
        'other'   => 'Другое',
        default   => $type,
    };
}));
```

Использование в шаблоне:

```twig
{{ obs.object_type | format_type }}
{# 'galaxy' → 'Галактика' #}
```

## Контрольные вопросы

### 1. В чём отличие нативных PHP-шаблонов от Twig? Преимущества и недостатки каждого подхода

**Нативные PHP-шаблоны** — обычные `.php`-файлы, в которых PHP-код перемежается с HTML.

Преимущества: не требуют установки дополнительных библиотек, полный доступ ко всем функциям PHP, нет накладных расходов на компиляцию шаблонов.

Недостатки: ничто не мешает написать в шаблоне бизнес-логику — обращение к базе данных, сложные вычисления. Дисциплина разделения держится только на договорённости разработчика с самим собой. Нет автоматического экранирования — `htmlspecialchars()` нужно писать вручную везде.

**Twig** — отдельный язык шаблонов со своим синтаксисом (`{{ }}`, `{% %}`).

Преимущества: автоматическое экранирование всех переменных защищает от XSS по умолчанию. Наследование шаблонов (`extends`, `block`) позволяет не дублировать макет. Синтаксис намеренно ограничен — нельзя написать сложную логику прямо в шаблоне. Кастомные фильтры позволяют расширять возможности.

Недостатки: требует установки через Composer, добавляет зависимость в проект. Синтаксис нужно изучать отдельно.

### 2. Зачем разделять логику и представление? Какие проблемы возникают если их смешивать

Когда логика и HTML в одном файле — при изменении внешнего вида приходится редактировать тот же файл где находится обработка данных, и наоборот. Это увеличивает риск случайно сломать что-то несвязанное.

Конкретные проблемы при смешивании: тяжело тестировать логику отдельно от HTML; один и тот же макет приходится копировать в каждый файл и при изменении шапки сайта редактировать все файлы; шаблон становится нечитаемым когда между тегами стоят длинные блоки PHP-кода с запросами к базе.

После разделения логика в `index.php` заканчивается вызовом `render()` — шаблон получает уже готовые данные и только выводит их.

### 3. Что такое наследование шаблонов в Twig? Как работают `{% extends %}` и `{% block %}`

Наследование позволяет определить общую структуру страницы один раз в базовом шаблоне, а в дочерних шаблонах заменять только нужные части.

`layout.twig` объявляет блок-заглушку:

```twig
{% block content %}{% endblock %}
```

`list.twig` расширяет макет и заполняет блок своим содержимым:

```twig
{% extends 'layout.twig' %}

{% block content %}
    ...таблица наблюдений...
{% endblock %}
```

При рендеринге Twig берёт `layout.twig` целиком и вставляет содержимое блока `content` из `list.twig` на место заглушки. Навигация, стили, шапка — всё это в `layout.twig` и автоматически присутствует на каждой странице без копирования.
